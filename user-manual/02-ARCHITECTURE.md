## 2.0 设计哲学：为什么需要 MPM？

### 2.0.1 Vibe Coding 的困境

"Vibe Coding" 是 AI 时代的新范式：你用自然语言描述需求，LLM 帮你写代码。这种模式让编程效率提升了 10 倍，但也带来了一个**致命的信任危机**——**黑箱问题**：

| 阶段 | 传统开发 | Vibe Coding |
|------|----------|-------------|
| **计划** | 开发者思考后制定方案 | LLM 内部推理，用户不可见 |
| **执行** | 开发者逐行写代码 | LLM 批量生成，用户不可控 |
| **记忆** | 开发者记得做过什么 | LLM 每次对话从零开始 |
| **审计** | Git 记录每一行变更 | LLM 的决策过程无法追溯 |

### 2.0.2 MPM 的终极使命

> **让 Vibe Coding 从「闭眼信任」变成「睁眼协作」。**

MPM 围绕五个核心维度构建可靠性保障：

| 维度 | 用户痛点 | MPM 解决方案 |
|------|----------|--------------|
| **可预测** | "LLM 的编码风格每次都不一样" | `initialize_project` 自动分析项目风格，生成规则文件 |
| **可观测** | "不知道 LLM 在想什么" | `manager_analyze` 输出【情报包】，展示代码锚点 |
| **可追溯** | "找不到之前做了什么改动" | `memo` 记录每次操作，同步到 `dev-log.md` |
| **可控制** | "任务做到一半中断了" | `manager_create_hook` 创建断点，支持断点续传 |
| **上下文提取** | "LLM 靠猜测修改代码" | `code_search` 精确符号定位，`code_impact` 清洗影响上下文，节省 95% Token |

## 2.2 核心模块说明

### 2.2.1 Server 入口 (cmd/server/main.go)

**职责**：MCP 协议的入口点，负责：

1. **工具注册**：通过 `RegisterXXXTools()` 函数加载所有工具
2. **请求路由**：将 LLM 的工具调用请求分发到对应的处理函数
3. **协议对齐**：在首次请求时从 IDE 获取真实的项目路径
4. **服务发现**：启动心跳服务，向 HUD 注册实例

**关键组件**：

MCPServer 包含以下字段：
- `server`: MCP 协议服务器实例
- `sessionMgr`: 会话管理器，维护跨调用状态
- `astIndexer`: AST 索引器，用于代码分析
- `discovery`: 服务发现组件，用于 HUD 通信

### 2.2.2 会话管理 (tools/system_tools.go)

**职责**：维护跨调用的上下文状态。

**SessionManager 结构**包含以下字段：
- `ProjectRoot`: 当前绑定的项目根目录路径
- `Memory`: 记忆层实例，用于数据库访问
- `TaskChains`: 任务链映射，存储正在执行的任务
- `WikiSessions`: Wiki 会话映射，用于文档生成任务
- `Discovery`: 服务发现实例，向 HUD 注册心跳

### 2.2.3 数据库层 (core/database.go)

**职责**：提供项目级的 SQLite 数据库访问。

**数据库位置**：`<项目根目录>/.mcp-data/mcp_memory.db`

**核心表结构**：

| 表名 | 用途 |
|------|------|
| `memos` | 原子操作备忘（同步到 dev-log.md） |
| `known_facts` | 项目级已知事实（铁律、避坑） |
| `pending_hooks` | 待办钩子（任务断点） |
| `system_state` | 系统状态（如当前人格） |

**特点**：
- **WAL 模式**：启用 SQLite Write-Ahead Logging
- **项目隔离**：每个项目独立维护 `.mcp-data/` 目录

### 2.2.4 记忆层 (core/memory.go)

**职责**：对数据库层的高级封装。

### 2.2.5 数据模型 (core/models.go)

**职责**：定义数据库使用的结构体。

包括：
- `Memo`: 备忘录结构
- `Fact`: 事实结构
- `Hook`: 钩子结构
- `Task`: 任务结构

### 2.2.6 技术栈检测 (core/detector.go)

**职责**：项目技术栈检测（代码基因感知）。

自动识别项目使用的语言、框架和工具链。

## 2.3 记忆层接口 (core/memory.go)

**MemoryLayer 结构**包含：
- `db`: 数据库管理器实例
- `root`: 项目根目录路径

**核心方法**：

备忘录管理：
- `AddMemo()`: 添加备忘录
- `GetMemos()`: 获取备忘录列表
- `SearchMemos()`: 搜索备忘录

事实管理：
- `AddFact()`: 添加已知事实
- `QueryFacts()`: 查询事实

Hook 管理：
- `CreateHook()`: 创建待办钩子
- `ListHooks()`: 列出钩子
- `ReleaseHook()`: 释放钩子

## 2.4 数据流

### 典型任务执行流程

```
用户输入 "帮我找到 manager_analyze 函数"
         │
         ▼
┌─────────────────────────────────────────────┐
│ 1. IDE 调用 manager_analyze 工具            │
└─────────────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────────┐
│ 2. Manager 分析任务                         │
│    - 识别意图类型 (DEVELOP)                 │
│    - 加载 Known Facts & Hooks               │
│    - 生成【任务简报】                       │
└─────────────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────────┐
│ 3. LLM 调用代码定位工具                     │
│    - code_search(query='符号名')            │
└─────────────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────────┐
│ 4. Rust AST Indexer 执行搜索                │
│    - 解析代码结构                           │
│    - 返回符号位置和调用关系                 │
└─────────────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────────┐
│ 5. 结果返回并记录                           │
│    - 调用 memo 记录操作                     │
│    - 同步到 dev-log.md                      │
└─────────────────────────────────────────────┘
```

## 2.5 辅助系统

### 2.5.1 HUD (Heads-Up Display)

**位置**：`internal/services/cockpit_hud/`

**技术栈**：Rust + Tauri

**功能**：
- 显示当前活跃的 MCP Server 实例
- 显示项目路径和人格状态
- 显示待办钩子数量

**通信方式**：
- 读取 `~/.mcp-cockpit/servers.json` 获取实例列表
- 心跳机制：MCP Server 每 5 秒更新一次

### 2.5.2 Skill 系统

**位置**：`skills/`

**功能**：
- 为特定领域任务提供专家指南
- 兼容 Claude Code 官方 Skill 规范

**文件格式**：Markdown with YAML Frontmatter

```markdown
---
name: frontend-design
description: 前端设计指南
---

# 设计原则
...
```

### 2.5.3 dev-log.md

**位置**：项目根目录

**功能**：
- 短期记忆窗口（最近 100 条操作）
- 新对话时快速获取上下文
- 由 `memo` 工具自动维护
- 与 `.mcp-data/mcp_memory.db` 中的 `memos` 表并行：每次写入 memo 时，还会在 `dev-log-archive/memo_archive.jsonl` 中追加一条 JSON 记录，作为可重放的物理备份，用于数据库灾难恢复

## 2.7 设计原则

MPM 遵循以下设计原则：

1. **单一职责**：每个工具只负责特定领域
2. **延迟初始化**：数据库和服务按需创建
3. **项目隔离**：每个项目有独立的数据库
4. **协议对齐**：主动从 IDE 获取真实的工作区路径
5. **两阶段执行**：先分析、后执行
6. **幂等设计**：重复调用同一工具不会产生副作用
